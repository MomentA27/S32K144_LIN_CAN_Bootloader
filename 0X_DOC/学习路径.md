# S32K144 LIN Bootloader 学习路径

## 前言

本学习路径面向从零开始构建 S32K144 LIN Bootloader 的开发者。以当前工程代码为参考，按从底层到上层、从协议到实现的顺序，循序渐进地掌握 LIN Bootloader 开发的完整技术栈。

每个阶段包含：学习目标、核心知识点、产出物、可参考的现有代码文件。

## 阶段五：LIN 传输层（ISO 17987-2）

### 学习目标
实现 ISO 17987-2 LIN 传输层协议，能进行多帧数据的分段传输与重组。

### 核心知识点
- LIN TP 与 CAN TP (ISO 15765-2) 的差异（无流控帧 FC）
- PCI 字节格式：高 4 位 = 帧类型，低 4 位 = 数据长度或序号
- 帧类型：
  - **SF（Single Frame）**：≤6 字节数据，单帧完成
  - **FF（First Frame）**：≥7 字节数据，首帧携带总长度
  - **CF（Consecutive Frame）**：后续帧，每帧 ≤6 字节，SN 递增
- N_PCI 参数与定时：
  - N_As / N_Ar：发送/接收单帧超时
  - N_Bs / N_Br：发送方/接收方首帧到连续帧的间隔
  - N_Cs / N_Cr：连续帧之间的最小/最大间隔
  - STmin：连续帧最小间隔
  - BlockSize：每块包含的 CF 数量
- 序列号（SN）管理：SN 从 0 开始，0~15 循环，不连续视为错误
- 超时处理与错误恢复机制
- LIN 诊断地址：NAD（Node Address）概念，物理寻址 vs 功能寻址 vs 广播

### 实践任务
1. 画出 LIN TP 的完整状态转换图（8 个状态）
2. 实现 SF 单帧的发送与接收（≤6 字节数据）
3. 实现 FF + CF 多帧的发送（总数据 > 6 字节，首帧发长度 + 前 7 字节，后续帧发剩余数据）
4. 实现 FF + CF 多帧的接收与数据重组
5. 实现 N_Cr 超时检测与错误处理
6. 实现 SN 序号校验（不连续时报告 N_WRONG_SN）
7. 编写多帧收发测试（发送 50 字节数据，验证接收端正确重组）

### 产出物
- 完整的 LIN TP 状态机模块（可独立于 Bootloader 测试）
- 状态转换图（建议用 draw.io 或手绘后拍照）

## 阶段六：UDS 诊断协议（ISO 14229）

### 学习目标
理解 UDS 协议核心概念，能实现 Bootloader 所需的 4 个核心诊断服务。

### 核心知识点
- UDS 协议模型：客户端（Tester）↔ 服务端（ECU）
- 诊断消息格式：SID + SubFunction + Data
- 响应格式：正响应（SID + 0x40）vs 负响应（0x7F + SID + NRC）
- 否定响应码（NRC）：常用错误码含义
- **Bootloader 核心服务（4 个）**：
  - **0x10 DiagnosticSessionControl**：会话切换（Default / Programming / Extended）
  - **0x27 SecurityAccess**：安全解锁（Seed & Key），AES-128 算法
  - **0x34/0x36/0x37 RequestDownload / TransferData / RequestTransferExit**：数据下载三步骤
  - **0x31 RoutineControl**：启动固件校验（CRC 校验 + 启动跳转）
- 辅助服务（按需）：
  - **0x3E TesterPresent**：保持诊断会话活跃，长耗时操作时维持会话
  - **0x22 ReadDataByIdentifier**：读取 ECU 信息（版本、Flash 大小等）
  - **0x2E WriteDataByIdentifier**：写入配置数据
  - **0x85 ControlDTCSetting**：编程期间关闭 DTC 记录
  - **0x28 CommunicationControl**：通信控制
- UDS 服务在 LIN TP 上的映射（USDT → Single Frame / FF-CF）
- 会话层安全状态机（Locked → Unlocked）

### 实践任务
1. 实现 0x10 会话控制（Default ↔ Programming 会话切换，含 P2 Server 时间上报）
2. 实现 0x27 安全访问（收到 Seed 请求 → 生成随机种子 → 发送给 Tester → 接收 Key → AES-128 验证）
3. 实现 0x34/0x36/0x37 下载流程（解析下载地址和大小 → 按块接收 → 临时缓存 → 校验通过后写入 Flash）
4. 实现 0x31 固件校验（计算 CRC → 比对 → 设置 APP 有效标志）
5. 实现 0x3E TesterPresent（处理会话保活请求）
6. 使用 CANoe / 周立功 LIN 工具 / 开源 LIN 工具模拟 Tester，测试完整 UDS 会话

### 产出物
- 完整的 UDS 诊断服务层模块
- 使用 LIN 诊断仪的完整 UDS 升级流程演示（Session Control → Security Access → Download → Checksum → Jump to APP）
- 一份 UDS 正响应/负响应的实际报文 log

## 阶段七：Flash 应用程序层

### 学习目标
实现完整的 Flash 编程状态机，支持 A/B 分区 FOTA、安全启动校验。

### 核心知识点
- Flash 下载状态机的 4 个阶段：
  - **FL_REQUEST_STEP**：解析 UDS 0x34 请求，获取下载地址和总大小
  - **FL_TRANSFER_STEP**：接收 UDS 0x36 数据块，累积到编程缓冲区
  - **FL_EXIT_TRANSFER_STEP**：接收 UDS 0x37 请求，确认传输完成
  - **FL_CHECKSUM_STEP**：计算已写入数据的 CRC，与预期比对
- Flash 操作调度状态：
  - `FLASH_IDLE` → `FLASH_ERASING`（擦除扇区） → `FLASH_PROGRAMMING`（写 Flash） → `FLASH_CHECKING`（校验） → `FLASH_WAITTING`
- 扇区管理：判断写入地址跨越哪些扇区、擦除粒度对齐
- 编程缓冲区：128 字节块写入（PROGRAM_SIZE），剩余字节用 0xFF 填充
- **A/B 分区 FOTA**：
  - APP_A 和 APP_B 两个应用槽位
  - 每个槽位的 APP 状态结构（CRC + Counter）
  - 启动时通过 Counter 比较 + CRC 校验选择启动最新有效 APP
  - 安全回退机制：新 APP 无效时回退到旧 APP
- 大扇区擦除时的超时处理：发送 UDS 0x3E TesterPresent 请求更多时间
- Flash 操作期间的中断保护（关键区禁用中断）
- Flash 驱动程序下载到 SRAM 执行

### 实践任务
1. 实现 Flash 下载状态机（Request → Transfer → Exit → Checksum 完整流程）
2. 实现 128 字节块编程，支持跨扇区写入
3. 实现 APP 状态结构（CRC + Counter），支持 A/B 双区
4. 实现启动时的 APP 选择逻辑（校验 CRC → 比较 Counter → 选择最新有效 APP → 跳转）
5. 测试：通过 UDS 下载一个完整 APP 到 Flash，验证跳转后 APP 正常运行
6. 测试回退：下载一半断电，验证 Bootloader 能检测并回退到旧 APP

### 产出物
- 完整的 Flash_app 模块
- 支持 A/B 双区 FOTA 的 Bootloader
- 断电恢复测试通过（下载中断后能安全恢复或回退）

## 阶段八：系统集成与主循环

### 学习目标
将所有模块串联为完整的 Bootloader，编译并烧录到 S32K144 实际运行。

### 核心知识点
- 主循环架构：状态机驱动 vs 事件驱动
- 模块间数据流：
  ```
  LIN 帧 → LIN TP 收 → FIFO → TP 层解包 → UDS 解析 SID
  → Flash_app 处理 → 写 Flash → UDS 组装响应 → TP 层打包
  → LIN TP 发 → LIN 帧发送
  ```
- 定时器驱动的周期性调用（`TP_MainFun` / `TP_SystemTickCtl`）
- 跳转到 APP 的流程：
  1. 校验目标 APP 的栈指针（SP）和复位向量（PC）的合法性
  2. 禁用全局中断
  3. 重置外设到复位状态
  4. 设置 MSP 为目标 APP 的栈顶地址
  5. 通过目标 APP 的复位向量跳转
- 启动判定逻辑：启动 → 检测更新标志 → 选择 APP → 判断是否进入 Bootloader 模式 → 如无需更新则跳转 APP

### 实践任务
1. 设计并实现主循环，串联 TP 收发包 + UDS 解析 + Flash 下载
2. 实现 APP 跳转函数（包括栈指针设置、外设复位）
3. 实现启动判定逻辑（检测是否需要进入 Bootloader 模式）
4. 编写一个简单的 APP（如 LED 不同频率闪烁），用于验证跳转
5. 完整刷写流程测试：Bootloader → UDS 下载 → Flash 写入 → 校验 → 跳转 → APP 运行
6. 重新上电测试：上电 → Bootloader 检测 → 直接跳转 APP（无新的更新请求时）

### 产出物
- 一个完整的、可在 S32K144 开发板上运行的 LIN Bootloader
- 配合测试 APP 的完整 LIN UDS 刷写（FOTA）演示
- 一份完整的数据流图（从 LIN 帧到 Flash 写入的完整路径）

### 推荐工具
- **逻辑分析仪**：Saleae Logic 或兼容品，抓取 LIN 总线波形
- **LIN 收发器**：TJA1020/TJA1021 模块
- **LIN 诊断仪**：周立功 LIN-USB 转换器，或使用 Arduino + MCP2004 自制
- **S32K144 开发板**：NXP S32K144EVB 或第三方兼容板
- **GDB 调试**：通过 PEMicro OpenSDA 使用 GDB Server 调试

### 重要提醒
- Flash 操作前务必备份原始固件，错误的擦除操作会永久损坏程序
- 不要在生产 ECU 上直接测试未验证的 Flash 操作代码，先在开发板上充分验证
- LIN TP 的无流控帧特性与 CAN TP(ISO 15765-2) 不同，注意区分，不要混用协议概念
- 当前代码注释均为中文，理解起来无障碍，建议对照 ISO 17987-2 标准文档一起阅读
